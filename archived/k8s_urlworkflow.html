Microservice Workflow in Kubernetes

A step-by-step guide to how a user request flows through Kubernetes components.

Step 1: User Sends a Request

The user accesses the application using a URL (e.g., https://myapp.example.com).
The request is sent over the internet to the Kubernetes cluster’s public endpoint.

Step 2: DNS Resolves the Domain

A DNS server resolves the domain name (myapp.example.com) into the public IP of the cluster’s Load Balancer.
Tools like Azure DNS, AWS Route 53, or Google Cloud DNS are used for domain resolution.

Step 3: Load Balancer (Layer 4)

A Layer 4 (L4) Load Balancer forwards TCP/UDP traffic to Kubernetes nodes hosting the Ingress Controller.
Examples include Azure Load Balancer and AWS ELB.

Step 4: Ingress Controller (Layer 7)

The Ingress Controller handles HTTP/HTTPS traffic, routing it based on hostnames or paths as defined in Ingress resources.
SSL/TLS termination occurs at this layer for secure communication.

Example Ingress Resource

apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: myapp-ingress
spec:
  rules:
  - host: myapp.example.com
    http:
      paths:
      - path: /api
        pathType: Prefix
        backend:
          service:
            name: backend-service
            port:
              number: 8080
Step 5: Kubernetes Service

The Kubernetes Service forwards traffic to pods based on labels.
Service types like ClusterIP, NodePort, and LoadBalancer define traffic exposure.

Example Service Configuration

apiVersion: v1
kind: Service
metadata:
  name: backend-service
spec:
  selector:
    app: backend
  ports:
  - protocol: TCP
    port: 8080
    targetPort: 8080
Step 6: Pods Process the Request

Pods host containers that execute the application logic. They process the incoming requests and, if necessary, interact with other services (e.g., databases).

Step 7: Inter-Service Communication

Microservices within Kubernetes communicate using internal DNS (e.g., http://database-service:3306).
Service meshes like Istio or Linkerd enable retries, traffic shaping, and mTLS communication.

Step 8: Response Flows Back

The processed response is sent back through the layers:
Pod → Service → Ingress Controller → Load Balancer → User.
The user sees the result in their browser or client.
